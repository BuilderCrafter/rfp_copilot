# Assessment Client Information Request Matrix

## Purpose

This matrix defines what the client needs to provide before Northstar can make a reliable bid decision, price the work, and prepare a responsible proposal. It is synthetic demo content for initial RFP assessment.

The assessment should use this document to populate missing-information checklist items. These are buyer-side information gaps, not materials Northstar must submit.

## Core Client Inputs Required

### Procurement Calendar

The client should provide:

- Proposal submission deadline.
- Clarification question deadline.
- Expected answer publication date.
- Shortlist date.
- Demo or presentation dates.
- Award date.
- Expected contract signature date.
- Desired project start date.
- Target go-live date.
- Hard regulatory, political, seasonal, funding, or operational deadlines.

Missing or partial calendar information should become a high-severity clarification item because it affects proposal effort, staffing, and delivery feasibility.

### Evaluation Criteria

The client should provide:

- Technical and commercial scoring weights.
- Mandatory pass/fail criteria.
- Minimum qualification criteria.
- Shortlist process.
- Demo, interview, or presentation expectations.
- Reference-check process.
- Requirements for compliance matrices.
- Whether price is lowest-cost, best-value, or quality-weighted.

If evaluation criteria are absent, the assessment should request scoring weights and award decision process. This affects bid/no-bid because a price-dominant procurement may be a weaker fit.

### Budget and Commercial Guardrails

The client should provide one or more of:

- Budget range.
- Estimated contract value.
- Funding source.
- Maximum contract value.
- Price ceiling.
- Target implementation budget.
- Expected annual recurring budget.
- Preferred pricing model.
- Whether discovery can be separately priced.
- Whether phased delivery or optional scope is allowed.

If no budget is disclosed, ask for affordability guardrails. Do not treat missing budget alone as no-bid, but mark it as a commercial risk.

### Scope and Outcomes

The client should provide:

- Business problem.
- Desired outcomes.
- Target users.
- User groups and personas.
- Current process pain points.
- In-scope functions.
- Out-of-scope functions.
- Service channels.
- Expected transaction volumes.
- Reporting and analytics needs.
- Success measures.

If scope is vague, request workflow examples, use cases, process diagrams, user roles, service categories, and expected volumes.

### Functional Requirements

The client should provide:

- Detailed workflow requirements.
- Role and permission model.
- Administrative functions.
- Content management needs.
- Notification channels.
- Document upload requirements.
- Approval, review, escalation, and exception handling.
- Reporting and dashboard needs.
- Localization and accessibility expectations.
- Mobile and browser support requirements.

If the RFP only states broad goals, ask for must-have workflows and acceptance examples.

### Integration Information

The client should provide:

- System inventory.
- Integration owners.
- API documentation.
- Sample payloads.
- Authentication methods.
- Rate limits.
- Environment availability.
- Test credentials or sandbox plan.
- Error-handling expectations.
- Retry and reconciliation expectations.
- Batch versus real-time requirements.
- Data ownership and operational support model.
- Third-party vendor contact process.

Missing integration detail is high severity when the RFP asks for multiple integrations or fixed-price delivery.

### Data Migration Inputs

The client should provide:

- Source systems.
- Record counts.
- Data formats.
- Data dictionaries.
- Data quality assessment.
- Historical data retention requirements.
- Cleansing ownership.
- Mapping rules.
- Migration validation criteria.
- Test data availability.
- Cutover expectations.
- Archive and deletion rules.

If the client requests migration but gives no data detail, ask for source extracts, record volumes, data quality constraints, and cleansing responsibilities.

### Security and Privacy Inputs

The client should provide:

- Data classifications.
- Applicable regulations.
- Privacy obligations.
- Security questionnaire.
- Data processing agreement.
- Required technical controls.
- Identity and access model.
- Logging and audit requirements.
- Encryption expectations.
- Vulnerability management expectations.
- Penetration testing requirements.
- Incident notification timeframes.
- Data residency requirements.
- Retention and deletion requirements.
- Subprocessor rules.

If the RFP includes sensitive personal data, health data, financial data, government data, or identity documents, missing security/privacy inputs should be high severity.

### Legal and Contract Inputs

The client should provide:

- Draft contract.
- Terms and conditions.
- Liability cap.
- Indemnity language.
- Insurance requirements.
- Data processing agreement.
- Intellectual property terms.
- Acceptance terms.
- Termination rights.
- Liquidated damages.
- Penalty clauses.
- Audit rights.
- Source-code escrow or ownership requirements.

If the RFP asks the vendor to accept terms but does not provide terms, request them before final bid approval.

### Acceptance and Delivery Inputs

The client should provide:

- Acceptance criteria.
- UAT responsibilities.
- Defect severity definitions.
- Sign-off process.
- Test environment expectations.
- Training responsibilities.
- Handover requirements.
- Documentation requirements.
- Production readiness criteria.
- Warranty period.
- Support transition process.

If acceptance criteria are missing, mark a delivery risk because it affects fixed-price scope and go-live accountability.

## Due-Diligence Inputs From The Client

Some RFPs ask vendors to provide financial, corporate, or personnel evidence. The client should clarify:

- Whether a company credit rating is required, and which agency or format is acceptable.
- Whether audited financial statements are required, and for which fiscal years.
- Whether a bank reference letter, performance bond, parent guarantee, or financial standing certificate is required.
- Whether employee criminal background checks are required.
- Whether employee credit checks are required.
- Whether government security clearance, right-to-work checks, drug testing, or personnel screening attestations are required.
- Which roles are subject to screening.
- Jurisdictions where checks must be performed.
- Required timing of checks.
- Whether summary attestations are acceptable instead of individual reports.
- Privacy, consent, retention, and disclosure requirements for personnel checks.

If due-diligence requirements are vague, the assessment should create missing-information items for clarification. Sensitive personnel screening should not be assumed.

## Client Information Severity Guide

High severity:

- Missing timeline, evaluation criteria, legal terms, critical integrations, security/privacy obligations, or fixed-price scope details.

Medium severity:

- Missing budget, data migration detail, acceptance criteria, support transition, or optional attachments.

Low severity:

- Minor formatting, portal upload metadata, branding instructions, or optional appendix preferences.

Critical severity:

- Missing information that makes bid submission legally unsafe, impossible to price, impossible to staff, or likely outside company risk appetite.

## Example Clarification Requests

- Please provide the scoring model, mandatory pass/fail criteria, shortlist process, and demo expectations.
- Please confirm the available budget range or maximum contract value.
- Please share the full integration inventory, API documentation, sample payloads, and integration owner contacts.
- Please provide source data volumes, formats, data dictionaries, data quality assumptions, and migration validation criteria.
- Please provide draft contract terms, liability cap, indemnity language, insurance requirements, DPA, and any penalty clauses.
- Please confirm whether company credit-rating evidence, audited financial statements, bank references, performance bonds, or guarantees are required.
- Please confirm whether employee criminal background checks, employee credit checks, or security clearances are required and whether vendor attestations are acceptable.
- Please define acceptance criteria, UAT responsibilities, defect severity rules, sign-off process, and warranty expectations.
